# p4 — Production validation evidence (memory governance)

Real runs against a real Weaviate (ephemeral `cr.weaviate.io/semitechnologies/weaviate:1.36.2`,
BYO-vectors, `--rm`, torn down after). Branch retrieval/ingest code. **No mocks, no unit tests** —
production reads/writes against a live vector store. Supports the p6 audit's evidence requirement.

## RUN A — read-side exclusion works (V1 + V2)
Wrote a current tile (`is_superseded=false`) and a superseded tile (`is_superseded=true`); queried
with the filter clauses generated by the **branch code** (`retrieval._build_where_filter` /
`retrieval_v2._build_filter`):

```
V1 default (exclude superseded):  ['current doc D v2']                          # superseded EXCLUDED
V1 opt-out  (include_superseded):  ['current doc D v2', 'superseded doc D v1']   # both
V2 default (exclude superseded):  ['current doc D v2']                          # superseded EXCLUDED
no filter (sanity):               ['current doc D v2', 'superseded doc D v1']   # both present
```
The boolean `{is_superseded NotEqual true}` predicate (which replaced the empty-string predicate that
*errors* on a real Weaviate) correctly excludes superseded tiles by default and includes them on opt-out.

## RUN B — supersede is fail-loud / fail-closed (BLOCKER-1 fix)
Probed `ISMACore._find_superseded_tile_ids` against a **real dead Weaviate** (`127.0.0.1:9`,
connection-refused):
```
PASS — fail-loud RuntimeError raised:
  supersede lookup unreachable (http://127.0.0.1:9/v1/graphql): HTTPConnectionPool(... port=9): Max retries exceeded
```
Before the fix it returned `[]` silently → the new version would commit while the prior stayed visible
(a zombie). Now it raises; the supersede runs *before* the new-tile write, so the write aborts cleanly
(no zombie, no half-superseded state) and surfaces as an embed failure.

## RUN C — graceful degradation verified (resolves BEIR no-regression)
Wrote three tiles — `is_superseded=false`, **no `is_superseded` property at all**, and
`is_superseded=true` — and queried the default filter:
```
{is_superseded NotEqual true}  ->  ['FALSE tile', 'NOFLAG tile']     # TRUE excluded; null/absent SURVIVES
```
So `NotEqual true` matches tiles that **lack** the flag. Consequences (Observed):
- The "degrades gracefully — a tile lacking the flag stays visible" claim is TRUE.
- **BEIR SciFact no-regression by construction:** the filter only removes `is_superseded==true` tiles;
  every false/null tile passes. SciFact has no superseded tiles, so the added clause is a no-op → no
  regression. (Verified the filter *semantics* on a real store rather than re-running the full BEIR
  harness; the committed baseline `benchmarks/results/scifact.json` stands.)

## Deploy requirement — corrected (was overstated)
- **Schema-presence is MANDATORY:** the `is_superseded` property must exist in the `ISMA_Quantum`
  schema before the read-side filter serves, or queries error `"no such prop"` (observed on live
  `:8088`, which predates the property). A fresh store auto-creates it on first governed write; an
  existing store needs a one-time schema-add (one governed write, or an explicit schema POST).
- **A values-backfill is OPTIONAL** (earlier docs overstated it as required): RUN C shows un-flagged
  legacy tiles stay visible under the default filter, so they are not hidden. Backfill only matters if
  you want every tile to carry an explicit `false`.

## Residual (honest)
- Full ingest→supersede end-to-end through `isma_core.ingest()`'s queue pipeline was not driven here;
  the write-side stamping + supersede + fail-loud are source-verified and exercised at the function
  level (RUN B) + the tile/filter level (RUN A/C). The Weaviate write API used in RUN A/C is the same
  one the write path uses.
